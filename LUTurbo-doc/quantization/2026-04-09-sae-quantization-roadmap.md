# SAE 量化研究总体路线图

## 1. 文档目的

本文档不是某一个具体实验的实现设计，而是对当前 SAE 量化研究做一次整体拆解。

它主要回答四个问题：

1. 我们整体上准备分几步推进 SAE 量化；
2. 为什么要按这些步骤推进，而不是一步到位；
3. 每个阶段做完之后，可能会出现哪些结果；
4. 不同结果分别意味着什么，下一步应该怎么走。

这份文档的作用，是把后续所有单点实验串起来，形成一条清晰的研究主线。

## 2. 当前背景

截至目前，我们已经完成了多轮 `product_key_expert_jumprelu` SAE 的 8bit 量化仿真评估，结论比较稳定：

- `expert-only W8A8` 误差几乎可以忽略；
- `full-encoder W8A8` 误差仍然很小；
- `full-encoder + decoder W8` 和 `full-encoder + decoder W8A8` 也都表现良好；
- 对当前 SAE 主干而言，8bit 本身已经不是主要精度风险点。

这说明一个很重要的事实：

**当前工作的核心问题，已经不再是“SAE 主干能不能做 8bit”，而是“我们要不要继续把训练目标、部署分布和在线补偿也一起纳入量化体系”。**

因此，后续研究不能再只做零散的单点量化实验，而需要一条完整路线。

## 3. 总体目标

整体目标可以分成三个层次。

### 3.1 第一层目标：确认 SAE 主干的 8bit 边界

这个目标目前已经基本完成。我们已经知道：

- encoder 可以做 8bit；
- decoder 可以做 8bit；
- 稀疏激活 `top_acts` 也可以做 8bit；
- 在当前指标口径下，精度损失很小。

### 3.2 第二层目标：让训练目标与部署目标更一致

这是接下来的核心目标。

因为当前训练大体还是：

- 输入：BF16 激活
- SAE 参数：FP32 / BF16 训练
- 输出：BF16 重构

而如果最终部署希望更接近：

- 输入：int8
- SAE 主干：8bit 友好
- 输出：int8

那么训练阶段也应该逐步感知这种部署形态。

### 3.3 第三层目标：形成真实 CPU 可落地的低精度方案

最终目的不是只得到一套“仿真指标很好看”的量化方案，而是得到：

- CPU 上可运行；
- 真正高效；
- 与 LUTurbo 主链路兼容；
- 不明显损伤精度

的一整套低精度方案。

## 4. 为什么不能一步到位

从直觉上看，既然现在主干 8bit 已经很稳，一个自然想法是：

- 能不能直接上“全链路 int8”甚至“训练时就完全 int8”。

但从工程和研究角度看，这样做风险太大，原因有四个。

### 4.1 训练目标和部署目标目前还没有完全对齐

现在的 SAE 训练仍然是典型的浮点重构训练。

如果直接跳到“纯 int8 训练”，我们会同时改变：

- 输入分布；
- 输出目标；
- SAE 内部数值行为；
- 梯度近似方式；
- 评价口径。

这样即使结果变好或变坏，也很难判断到底是哪一部分在起作用。

### 4.2 当前还缺统一的评价口径

如果 int8 后 `FVU` 变小，有两种完全不同的解释：

- 任务本身变简单了，因为量化后的目标更粗；
- 模型真的更适配部署目标了。

如果没有统一的评估口径，很容易把前者误认为后者。

### 4.3 在线补偿还没有纳入量化路线

当前 SAE 主干已经很稳，但 LUTurbo 还有在线补偿阶段。

从系统角度看，最终瓶颈可能并不在 SAE 主干，而在：

- 在线补偿；
- 稀疏 gather；
- CPU 上真实 int8 kernel 形态。

所以不能只看 SAE 内部。

### 4.4 CPU 上低比特并不一定天然高效

在通用 CPU 上，真正成熟、稳定、通用的低精度路径通常还是 int8。

更低 bit（比如 int4 / int2）即使理论上更省存储，也未必更高效，因为：

- pack/unpack 有额外开销；
- 稀疏/小粒度算子不一定吃到低比特 kernel；
- 在线补偿和按列 gather 这类路径常常不是标准大 GEMM。

所以路线必须是分阶段、逐层验证的。

## 5. 总体阶段划分

建议将 SAE 量化研究分成六个阶段。

## 5.1 Phase 0：统一评估口径

### 这一阶段做什么

先不改训练，只把后续所有实验的比较口径统一下来。

建议至少固定以下指标：

- `FVU_fp_teacher`
- `FVU_deploy`
- `exceed_alpha_0.5_fp_teacher`
- `exceed_alpha_0.5_deploy`
- `quant_floor`

其中：

- `FVU_fp_teacher`：看对原始浮点目标的还原能力；
- `FVU_deploy`：看对量化部署目标的还原能力；
- `quant_floor`：看单纯量化本身带来的误差下界。

### 为什么必须先做这一步

因为后面所有 I/O 量化训练、QAT、部署对齐训练都会改变输入输出形态。

如果不先统一评价口径，后面的结果就很容易失真。

### 这一阶段可能得到什么结果

#### 结果 A：`FVU_deploy` 显著优于 `FVU_fp_teacher`

这通常说明：

- 部署世界里的目标更简单；
- int8 任务比原始 BF16 任务更容易拟合。

这不是坏事，但不能简单解读成“模型更强了”。

#### 结果 B：两者接近

这说明量化前后的任务难度变化并不大，后续 QAT 的收益可能也不会特别大。

#### 结果 C：`quant_floor` 已经很高

这说明输入/输出一旦量化，本身就损失了较多信息。后续训练时量化的潜在收益可能主要是“适应不可逆损失”，而不是追求比 BF16 更好的重构。

### 这一步做完之后意味着什么

这一步做完，后续所有实验都有了统一的解释框架。它是整个路线的度量基础。

## 5.2 Phase 1：输入输出量化感知训练（I/O QAT）

### 这一阶段做什么

把训练链路改成：

```text
x_fp -> fake_quant_in -> SAE -> fake_quant_out -> loss
```

这一阶段只要求训练显式感知：

- 输入会经过 int8 量化；
- 输出最终也会经过 int8 量化。

但 SAE 参数、梯度、优化器状态仍然保持浮点。

### 为什么先做这一步

因为这是从“浮点训练”到“部署友好训练”的最小改动版本。

它能回答一个非常关键的问题：

**如果训练时只显式模拟 int8 I/O，模型是否就已经能明显更适应部署形态？**

如果答案是肯定的，说明没必要一上来就做大改动。

### 这一阶段可能得到什么结果

#### 结果 A：`FVU_deploy` 明显改善，`FVU_fp_teacher` 基本不变

这是最理想的情况。

说明：

- 训练看到 int8 I/O 后确实更适应部署形态；
- 但又没有明显损伤原始语义；
- I/O QAT 是值得继续推进的。

这时下一步应该进入 `Phase 2`。

#### 结果 B：`FVU_deploy` 改善，但 `FVU_fp_teacher` 变差明显

说明模型开始更偏向部署目标，而偏离原始浮点 teacher。

这时需要判断：

- 部署优先级是否高于 teacher 对齐；
- 双目标 loss 的权重是否需要调回更保守。

#### 结果 C：两者都没明显改善

说明只做 I/O fake quant 的收益有限。

这意味着：

- 要么当前 PTQ 已经足够好；
- 要么真正的瓶颈不在 I/O，而在 SAE 内部主干。

这时可以考虑直接进入 `Phase 2`，或者干脆继续以 PTQ 为主。

#### 结果 D：训练明显变不稳定

说明即便只是 I/O fake quant，当前配置也比较敏感。

这时需要先稳住：

- clipping；
- scale 设计；
- loss 权重；
- 学习率。

### 这一步做完之后意味着什么

它会告诉我们：训练时量化到底有没有必要，以及“最便宜的一步”到底值不值得做。

## 5.3 Phase 2：SAE 主干 Partial QAT

### 这一阶段做什么

在 Phase 1 成功的基础上，逐步把 SAE 主干中的关键线性层也纳入 fake quant。

建议顺序：

1. `W_dec`
2. `expert_encoders`
3. `left_router` / `right_router`

### 为什么是这个顺序

因为当前已有 PTQ 结果表明：

- decoder 很稳；
- expert encoder 也很稳；
- router 相对更敏感。

因此最合理的推进顺序是从低风险到高风险。

### 这一阶段可能得到什么结果

#### 结果 A：Partial QAT 明显优于 PTQ

说明训练时显式适配量化误差是有价值的，尤其对主干内部模块有效。

这时应继续推进更完整的部署对齐。

#### 结果 B：只对个别模块有效

例如：

- `W_dec` 做 QAT 有收益；
- `router` 做 QAT 收益很小甚至有害。

这说明后续不需要追求“所有模块都 QAT”，而应该选择性使用。

#### 结果 C：和 PTQ 几乎没差别

说明当前主干已经足够 8bit 友好，继续在训练图里做更多量化未必划算。

这时可以把重点转回在线补偿或真实部署。

### 这一步做完之后意味着什么

它会回答：QAT 的收益到底来自哪里，是不是值得进入“更深入的量化训练阶段”。

## 5.4 Phase 3：部署分布对齐训练

### 这一阶段做什么

用量化 backbone 采样激活，再训练或 finetune SAE。

也可以进一步考虑：

- 输入来自 `W8A8 backbone`
- target 仍然对齐 `BF16 teacher`
- 或使用混合目标

### 为什么需要这一步

因为即使 SAE 自己量化得很好，如果最终部署时它接到的是量化 backbone 的激活，那么训练分布和部署分布之间仍可能存在失配。

### 这一阶段可能得到什么结果

#### 结果 A：明显提升部署指标

说明训练分布失配是真问题，部署对齐训练值得保留。

#### 结果 B：变化很小

说明 backbone 量化带来的分布漂移不大，这一步优先级可以降低。

#### 结果 C：指标变差

说明部署分布里可能混入了量化噪声，直接拟合它反而损伤了 teacher 对齐。

这时需要考虑更保守的混合目标，而不是纯部署对齐。

### 这一步做完之后意味着什么

它会告诉我们：未来如果真的把整套系统都部署成量化版本，SAE 是否必须做分布对齐训练。

## 5.5 Phase 4：在线补偿量化

### 这一阶段做什么

在 SAE 主干量化和训练路线比较明确之后，单独评估在线补偿是否也值得量化。

重点不是只看精度，而是看两件事：

1. 在线补偿在主干量化后是否成为新的 runtime 瓶颈；
2. 量化后是否还能保留大部分补偿收益。

### 为什么它不能太早做

因为在线补偿是“精度保险丝”。

如果在主干量化路线还不清楚的时候就去量化补偿，很容易混淆问题来源。

### 这一阶段可能得到什么结果

#### 结果 A：量化后精度几乎不受影响，而且运行占比又很高

这说明在线补偿非常值得量化。

#### 结果 B：量化后补偿收益丢失明显

说明在线补偿不适合作为通用 8bit 路径，更适合继续保留为浮点保险丝。

#### 结果 C：量化后即便可行，但运行占比很低

说明从系统优化角度看，收益不大，不必优先推进。

### 这一步做完之后意味着什么

它会回答：最终是不是要追求“连在线补偿也 8bit”，还是让它继续作为精度保护路径存在。

## 5.6 Phase 5：真实 CPU 部署验证

### 这一阶段做什么

将前面所有“仿真正确”“训练有效”的结果，真正落到 CPU 部署环境里。

关注的重点包括：

- int8 kernel 是否成熟；
- pack/unpack 代价是否可以接受；
- 稀疏 gather 路径是否真的高效；
- 在线补偿是否破坏整体收益。

### 为什么这一阶段必须放到最后

因为真实 CPU 部署的噪声因素太多。

如果在前面方案还不稳定时就直接做 CPU 优化，很容易花大量时间在低层细节上，却不知道高层路线本身是否正确。

### 这一阶段可能得到什么结果

#### 结果 A：精度和速度都成立

这是最终目标，说明路线闭环成立。

#### 结果 B：精度成立，但速度收益不明显

说明问题主要不在量化正确性，而在：

- kernel 选择；
- 稀疏访存模式；
- 系统实现细节。

#### 结果 C：速度有收益，但精度不稳

说明系统层面是对的，但量化训练/补偿/阈值策略还需要回头修正。

### 这一步做完之后意味着什么

它最终回答：这条路线是不是一条真正可交付的 CPU 推理方案。

## 6. 当前最合理的推进顺序

基于已有实验结果，当前最合理的顺序不是直接跳到“全链路 int8”，而是：

1. 统一评估口径；
2. 做 `I/O QAT`；
3. 看是否值得做 `Partial QAT`；
4. 再决定要不要做部署分布对齐；
5. 然后再处理在线补偿；
6. 最后做真实 CPU 部署验证。

这条路线的好处是：

- 每一步都在回答一个明确问题；
- 每一步失败了也知道问题出在哪；
- 不会把太多变量同时混在一起。

## 7. 当前阶段判断

结合现有结果，可以给出一个很清楚的当前判断。

### 7.1 已经基本确认的事情

- SAE 主干路径是 8bit 友好的；
- decoder 继续量化也没有明显风险；
- 当前阶段不需要优先去碰 threshold / bias 这些小参数。

### 7.2 现在最值得回答的问题

当前最值得回答的问题有两个：

1. **训练时显式感知 int8 I/O，到底有没有价值？**
2. **在线补偿是否会成为主干量化后的新瓶颈？**

其中，问题 1 更适合先做，因为它更基础，也更便宜。

### 7.3 因此当前推荐路线

当前推荐路线是：

- 先启动 `Phase 1: I/O QAT`
- 不直接上纯 int8 训练
- 不直接上全链路 QAT
- 不过早把在线补偿卷进来

## 8. 总结

这条 SAE 量化路线的核心思想，不是“尽快把所有东西都量化”，而是：

**先确认主干 8bit 是否成立，再逐步让训练目标、部署分布和系统实现对齐。**

到目前为止，主干 8bit 已经基本成立。

接下来最合理的一步，是用最小代价验证：

**如果训练时显式模拟 `int8 input + int8 output`，SAE 是否会更适应我们真正想要的部署世界。**

如果这一步成立，后续的 Partial QAT、部署分布对齐和在线补偿量化，才有进一步推进的意义。
