## 1. 项目概述

**目标**：在 CPU 平台上实现 LLM 的低延迟推理。

**核心思路**：用基向量分解（Basis Vector Decomposition）将 LLM 推理中的向量-矩阵乘法转换为查表操作，以存储换计算，利用 CPU 的大内存容量优势。

**关键假设**：
- LLM 推理是 memory-bound 的，即访存开销 ≈ 性能。
- 因此：如果一个方案的总访存量没有减少，性能一定没有提升；即使访存量减少了，如果是随机访存模式，实际性能提升也可能受限。
- CPU 比 GPU 更适合随机访存模式（独立线程执行，无 warp divergence），且内存容量远大于 GPU 显存。

**论文标题**：LUTurbo: Near-Lossless, Ultra-Low Latency LLM Inference on CPUs

## 2. 当前方案

### 2.1 基向量分解方法

LLM 推理的核心计算是向量-矩阵乘法 y = x W，其中 x ∈ R^h 是输入激活，W ∈ R^{h × n} 是权重矩阵。

LUTurbo 维护一个基向量库 B = {b_1, b_2, ..., b_N}，每个 b_i ∈ R^h。对于输入 x，将其分解为 K 个基向量的线性组合：

```
x ≈ Σ_{i ∈ S} α_i · b_i
```

其中 S 是选中的 K 个基向量索引，α_i 是组合系数。

系统预先计算每个基向量与权重矩阵的乘积并存入查找表：

```
P_i = b_i · W,  i = 1, 2, ..., N
```

推理时，查表 + 加权求和：

```
ỹ = Σ_{i ∈ S} α_i · P_i
```

**复杂度变化**：在线计算从 O(h × n) 降为 O(K × n)。当 K << h 时，计算量和访存量大幅减少。

**方法的核心本质**：基向量分解将在线计算变成了查表。查表的访存开销直接取决于激活的基向量数量 K（查表访存 = K × n）。同时，K 也影响重构精度——K 越小误差越大，需要在线补偿的维度越多，补偿访存 = p × n × h（每补偿一个输出维度 j，就要额外访问原始权重矩阵的一列 W_{:,j}，大小为 h；共 p×n 个维度需要补偿）。两者都可以统一为"访问了多少向量"：查表访问预计算表的行（每行 n 维），补偿访问原始权重矩阵的列（每列 h 维）。

因此，方案要可行，有两个缺一不可的条件：
1. **K 要足够小**（同时保证精度），以降低查表访存和总等效行访问数
2. **基向量选择算法的开销要足够低**，选择过程本身的访存不能抵消查表带来的收益

### 2.2 SAE 构建基向量库

使用稀疏自编码器（SAE）训练基向量库。

**SAE 与基向量分解的对应关系**：
- SAE 解码器矩阵 D ∈ R^{N × h} 的每一行 = 一个基向量 b_i
- SAE 稀疏中间表示 z ∈ R^N = 基向量组合系数
- SAE 重构过程 x̂ = z^T D = 基向量线性组合
- TopK 激活函数保证每个输入最多使用 K 个基向量

**训练细节**：
- 每层模型分别训练一个 SAE
- 训练数据：真实推理中收集的激活样本
- 损失函数：L = E[||x - x̂||²]
- 训练完成后，解码器矩阵直接作为该层的基向量库

### 2.3 在线补偿机制

基向量分解是近似计算，存在重构误差。LUTurbo 使用混合策略：

- 大多数输出维度：使用查表结果（便宜）
- 少数误差大的维度：执行精确在线计算 y_j = x · W_{:,j}（访问权重矩阵的一列，O(h)）

**阈值机制**：
- θ（theta）：离线绝对阈值。通过 Kneedle 算法在校准数据上分析每个算子的误差分布曲线，取拐点（elbow point）对应的误差值。每个算子各有一个 θ。
- τ（tau）：缩放系数（百分比），统一控制阈值的松紧程度。
- 实际阈值 = θ × τ。当某个输出维度的重构绝对误差 > θ × τ 时，对该维度执行在线精确计算。
- τ 越大 → 容忍误差越大 → 在线计算维度越少（p 越小） → 但模型精度损失越大。

**补偿比例 p**：需要在线计算的输出维度占总输出维度的比例。p 由 SAE 重构质量和 τ 共同决定。补偿计算复杂度为 O(p × h × n)。

**当前实验数据**：τ = 0.5 时，p ≈ 20%。但 τ = 0.5 意味着容忍较大的误差，对模型精度有损害。目标是将 τ 降到 0.2~0.3 的同时保持 p 在可接受范围，这需要通过提升 SAE 重构精度来实现。

### 2.4 完整推理 Pipeline

1. **基向量选择**：确定输入 x 应该使用哪 K 个基向量（索引集合 S）
2. **系数计算**：计算 K 个组合系数 {α_i}
3. **查表组合**：从查找表读取 K 个 P_i，计算加权和 ỹ = Σ α_i P_i
4. **在线补偿**：对误差超阈值的维度精确计算

其中步骤 3 和 4 的方案已确定，**步骤 1（基向量选择）是当前的核心瓶颈**。

### 2.5 关键参数及其关系

| 参数 | 含义 | 典型值 | 备注 |
|------|------|--------|------|
| h | 隐藏层维度 | 1024 ~ 8192 | 模型架构决定 |
| n | 权重矩阵输出维度 | 0.5h ~ 4h | 不同层不同 |
| N | 基向量库大小 | 8h | SAE 扩展倍数，实验确定 |
| K | 每个输入选择的基向量数 | ~128（h=1024 时） | 精度要求决定，K 越大精度越好 |
| θ | 离线绝对阈值 | 每算子不同 | Kneedle 算法从误差分布得出 |
| τ | 缩放系数 | 0.5（当前） | 目标降到 0.2~0.3 |
| p | 需要在线补偿的维度比例 | ~20%（τ=0.5 时） | τ 降低后 p 会上升，需 SAE 精度补偿 |

**关键约束**：
- N = 8h 是精度的下界，更小的 N 会导致不可接受的精度损失
- K ≈ 128（h=1024）是保证精度所需的最小值
- 任何方案的总访存量必须 < 原始 matmul 访存量 h × n，否则无性能提升

## 3. 核心瓶颈

项目目前面临两个相互关联的核心瓶颈：
1. **基向量选择的开销**（§3.1-3.2）：如何低开销地从 N 个基向量中选出 K 个
2. **SAE 重构精度**（§3.3）：如何让 SAE 用更少的 K 达到更低的重构误差

两者的关联：SAE 精度越高 → K 可以更小、p 可以更低 → 查表收益越大。但更好的 SAE 可能需要更大的 N，而更大的 N 又加重选择开销。

### 3.1 瓶颈一：基向量选择的开销

#### 3.1.1 问题本质

Pipeline 步骤 1（基向量选择）目前没有高效的实现方式。

选择过程可以分解为两个子问题：
1. **选择激活哪些基向量**（从 N=8h 个中选 K 个）
2. **确定它们的系数**（选定后计算内积，O(K × h)，开销可接受）

瓶颈在子问题 1。本质上是一个从大规模候选集中做 top-K 选择的问题。

#### 3.1.2 性能约束条件

在 memory-bound 假设下，**访存量 = 性能指标**。

原始 matmul 访存量：h × n（n = 0.5h ~ 4h）

任何基向量选择方案的访存量必须满足：

```
选择访存 + 系数计算访存 + 查表访存 + 补偿访存 < h × n
```

其中：
- 系数计算：K × h（≈ 128 × 1024 = 131K，可接受）
- 查表：K × n（≈ 128 × n，可接受）
- 补偿：p × h × n（≈ 0.25 × h × n，已经占了原始的 25%）

留给"选择"步骤的访存预算 = h × n - K × h - K × n - p × h × n

当 n = h 时：预算 ≈ h² - 128h - 128h - 0.25h² = 0.75h² - 256h ≈ 0.75h²

所以选择步骤的访存不能超过约 0.75 × h × n。看起来预算不小，但问题是现有方法都远超这个预算或接近原始开销。

**注意**：上述分析使用 p=0.25。当前实验中 τ=0.5 时 p≈0.2，与此接近。但若 τ 降到 0.2~0.3（为保证模型精度），p 会显著上升，留给选择步骤的预算会更小。

### 3.2 瓶颈二：SAE 重构精度

#### 3.2.1 问题本质

当前使用常规 SAE 架构（稠密编码器 + TopK 激活 + 解码器），重构精度不够高。具体表现：
- τ = 0.5（容忍较大误差）时，p ≈ 20%，性能可接受但模型精度受损
- 若将 τ 降到 0.2~0.3 以保证模型精度，p 会显著上升，削弱查表加速效果

SAE 精度问题的核心目标：**在给定 N（基向量库大小）的条件下，让 K 尽可能小、重构误差尽可能低，使得在较小的 τ（0.2~0.3）下 p 仍保持在 20%-30%。**

#### 3.2.2 当前 SAE 架构的局限

1. **编码器是稠密矩阵**：编码器 E ∈ R^{h × N}，计算 z = TopK(Ex + b) 需要完整的矩阵乘法。这既导致了训练出的基向量选择方式是稠密的（无法高效在线执行），也可能限制了重构质量的优化空间。

2. **架构/超参数未充分探索**：当前使用常规配置，可能存在更好的架构设计能够在相同 K 下达到更低的重构误差。

#### 3.2.3 SAE 精度与整体方案的关系

SAE 精度直接影响整体加速效果：

```
p=0.5 时：总访存 ≈ K×n + 0.5×h×n + 选择开销
         对比原始 h×n，即使选择开销为 0，加速上限也只有 ~2x

p=0.25 时：总访存 ≈ K×n + 0.25×h×n + 选择开销
          加速上限可达 ~4x（选择开销为 0 时）

p=0.1 时：总访存 ≈ K×n + 0.1×h×n + 选择开销
          加速上限可达 ~8x
```

因此，SAE 精度的提升对最终性能有倍数级影响，是与选择开销同等重要的瓶颈。

## 4. 已探索方案

### 4.1 ANNS 加速（不可行）

**思路**：将基向量选择转化为最大内积搜索（MIPS）问题，用近似最近邻搜索（ANNS）加速。

**失败原因**：
- ANNS 需要找 top-K，K=128 时搜索复杂度与 K 直接相关
- ANNS 的搜索开销随 K 增大而显著增长
- 加上 refine 步骤 O(K × h)，总开销接近甚至超过原始矩阵乘法
- 结论：当 K 较大时，ANNS 完全没有性能优势

### 4.2 直接使用 SAE 编码器（不可行）

**思路**：直接用 SAE 的编码器矩阵 E ∈ R^{h × N} 计算，取 top-K。

**失败原因**：
- 编码器矩阵大小为 h × N = h × 8h = 8h²
- 原始权重矩阵大小为 h × n，n 最大为 4h，即 4h²
- 编码器的访存量 8h² > 原始 matmul 的访存量 4h²
- 编码器本身就比模型权重还大，用它来做选择毫无意义
- 结论：访存量不减反增

### 4.3 讨论中的其他方向（未验证）

**子空间分解/乘积量化思路**：
- 将 h 维输入分成 M 个子空间，每个子空间独立维护小型基向量库（大小 C）
- 每个子空间独立做选择，搜索空间从 8h 降到 C
- 选择开销：M × (h/M) × C = h × C
- 问题：子空间间的交互信息丢失，可能影响重构精度
- 状态：未验证

## 5. 开放问题与探索方向

### 关于基向量选择开销

1. **选择步骤能否完全避免全局搜索？** 是否存在结构化的方法让基向量分配变成确定性映射？

2. **激活模式是否有可利用的规律？** 比如不同 token 激活的基向量是否有较大重叠？是否有聚类特征？

3. **能否用极小的模型做路由？** 比如训练一个远小于编码器的分类器来预测应该激活哪些基向量？

4. **两阶段选择是否可行？** 先用低开销方法粗选一批候选（比如 2K ~ 4K 个），再从中精选 K 个？

5. **基向量库结构能否利用？** 如果基向量之间存在层次结构或聚类结构，能否加速选择？

### 关于 SAE 精度

6. **是否有更优的 SAE 架构？** 比如多层编码器、残差连接、分组编码等，能否在相同 K 下降低重构误差？

7. **训练策略能否改进？** 比如课程学习（逐步减小 K）、辅助损失函数、更好的激活数据采样策略？

8. **K 和 N 的最优配比是什么？** 当前 N=8h, K=128(h=1024)，是否存在更好的 N/K 组合？

9. **两个瓶颈能否联合解决？** 是否存在一种 SAE 架构，其编码过程天然就是低开销的（即编码器本身就是稀疏/结构化的），同时保持高重构精度？

### 2026-03-16 | CG 最优系数 Phase 1 验证

- **方法**：保持 SAE 编码器选择不变，将系数计算替换为 CG 迭代求解（10 iter, fp32），对比内积系数的重构质量。额外用 `torch.linalg.lstsq` exact LS 确认 CG 收敛性
- **模型**：Qwen3-0.6B（18 实验）+ Qwen3-4B（16 实验），共 34 个实验覆盖多层多算子
- **结果**：CG(10 iter) = exact LS（gap=0%）。MSE reduction 10~17%，但 p_reduction@τ=0.5 仅 8~20%（绝对值 1~3pp）。o_proj MSE 改善最大（21~27%）但 p_reduction 最小（2~4%）
- **结论**：CG 系数确实优于内积，但推理时的额外计算开销（10 次 K×K matvec）可能抵消 p 的微小降低。**不建议进入 Phase 2**，CG 作为备选保留。详见 [实验文档](experiments/20260315-cg-coefficients-phase1.md)

### 2026-03-15 | ANNS 加速基向量选择

- **方法**：使用 ANNS 索引将基向量选择转化为 MIPS，加速 top-K 搜索
- **结果**：K=128 时 ANNS 搜索开销过大，无性能提升
- **结论**：不可行。当 K 较大时，ANNS 的复杂度与 K 直接相关，无法绕过

### 2026-03-15 | 直接 SAE 编码器

- **方法**：直接用编码器矩阵 E (h × 8h) 做矩阵乘法取 top-K
- **结果**：编码器访存 8h² > 原始权重访存（最大 4h²）
- **结论**：不可行。访存量不减反增，违反基本约束
