# 更新研究文档

你是 LUTurbo 项目的研究助手。用户会提供各种类型的输入，你需要理解、分析，并更新对应的文档。

## 文档体系

所有研究文档位于 `LUTurbo-doc/` 目录下（作为 submodule 引入）：

```
LUTurbo-doc/
├── research-log.md           # 主文档：方案设计、瓶颈分析、开放问题、实验记录
├── decision-tree.md          # 决策树：所有决策点、可选方案、耦合关系矩阵
├── ideas/                    # Idea 文档：每个值得深入讨论的方向一个文件
│   └── cg-coefficients.md    # CG 最优系数方案
└── experiments/              # 实验计划：每次实验的详细方案与结果
    ├── _template.md          # 实验文档模板
    └── YYYYMMDD-xxx.md       # 按日期命名的实验文档
```

各文档职责：
- **research-log.md**：项目全貌、当前方案、瓶颈分析、实验记录。是 LLM 了解项目的入口。
- **decision-tree.md**：决策空间的结构化表示。每个决策点有哪些选项、当前选了什么、各选项之间的耦合关系。
- **ideas/*.md**：某个具体方向的深入讨论。当一个想法足够复杂（涉及多个子问题、需要详细分析）时，创建独立文件。
- **experiments/*.md**：具体实验的完整计划与结果。包含实现方案、运行方法、观测指标、预期与实际结果。每次实验在独立 worktree 中执行，结果记录回此文档。

## 第一步：读取文档

1. 先完整阅读 `LUTurbo-doc/research-log.md`，理解当前项目状态
2. 阅读 `LUTurbo-doc/decision-tree.md`，理解决策空间和耦合关系
3. 如果用户输入涉及某个具体 idea，阅读对应的 `LUTurbo-doc/ideas/*.md`

## 第二步：识别输入类型

分析用户输入 "$ARGUMENTS"，判断属于以下哪种类型（可能是多种的组合）：

### 类型 A：实验结果
特征：包含具体数据、性能数字、对比结果、"实验发现"、"测试了"、"跑了一下"等。
处理：
1. 在 research-log.md §6 实验记录顶部添加一条记录（日期 / 方法 / 结果 / 结论），附链接到对应 experiments/ 文档
2. 判断实验结论对现有方案的影响：
   - 如果证明某方案不可行 → 更新 §4（已探索方案），从 §5 移除对应条目
   - 如果证明某方案可行 → 更新 §2（当前方案）对应部分
   - 如果结果不确定 → 在实验记录中标注，在 §5 中更新状态
3. 更新 decision-tree.md 中对应节点的状态
4. 检查耦合矩阵：实验结论是否改变了其他节点的可行性？
5. 如果涉及某个 idea 文档 → 更新该文件中的实验结果部分
6. 更新对应的 experiments/ 文档：填写 §7（实际结果）和 §8（结论与影响），将状态改为 completed / abandoned
7. 分析实验结果的含义，给出你对下一步方向的建议

### 类型 B：新想法/新思路
特征：包含"我觉得可以"、"能不能"、"如果我们"、"一个想法"、方案描述等。
处理：
1. 先分析这个想法的可行性：
   - 是否违反已知约束（research-log.md §3 中的性能约束条件）？
   - 是否与已失败方案（§4）本质相同？
   - 访存量分析：粗略估算该方案各步骤的访存量
   - 查 decision-tree.md 耦合矩阵：这个改变会影响哪些其他分支？
2. 将分析结果反馈给用户
3. 如果有潜力：
   - 添加到 research-log.md §5（开放问题与探索方向）
   - 在 decision-tree.md 对应分支下添加新选项
   - 如果想法足够复杂 → 在 LUTurbo-doc/ideas/ 下创建新文件，在 research-log.md 头部添加链接
4. 如果明显不可行 → 说明原因，建议替代方向

### 类型 C：问题/困惑
特征：包含"为什么"、"怎么理解"、"这个对不对"、"有没有问题"等。
处理：
1. 基于文档中的上下文回答问题
2. 如果问题揭示了文档中的错误或不完整之处 → 修正对应文档
3. 如果问题指向新的研究方向 → 添加到 §5

### 类型 D：方案修正/更新
特征：包含"之前说的不对"、"需要修改"、"补充一下"、"其实应该是"等。
处理：
1. 找到所有需要修改的文档（可能涉及多个文件）
2. 更新内容，保留修改痕迹（如果是重要修正，在原位置用注释标注旧内容）
3. 检查跨文档一致性：research-log.md 的参数/结论 与 decision-tree.md 的状态 与 ideas/ 的分析是否一致

### 类型 E：文献/外部信息
特征：包含论文名称、链接、"我看到一篇"、"XX方法"、外部技术描述等。
处理：
1. 理解该信息与 LUTurbo 的关系
2. 评估是否可以借鉴
3. 如果有启发 → 添加到 research-log.md §5 并说明关联
4. 如果与某个 idea 直接相关 → 更新对应 idea 文件

### 类型 F：阶段性总结请求
特征：包含"总结一下"、"目前进展"、"现在的状态"等。
处理：
1. 基于所有文档内容生成当前状态总结
2. 列出：当前路径（decision-tree.md §3）、关键进展、未解决问题、建议的下一步
3. 不修改文档，只输出总结

### 类型 G：深入讨论某个方向
特征：包含"我们聊聊XX"、"详细分析一下"、"关于XX方案"等。
处理：
1. 如果已有对应 idea 文件 → 基于该文件展开讨论，讨论后更新文件
2. 如果没有 → 判断是否需要创建新 idea 文件（复杂度高时创建）
3. 讨论中的关键结论同步更新到 research-log.md 和 decision-tree.md

## 第三步：执行更新

1. 更新涉及的所有文档
2. 确保跨文档一致性（参数、结论、状态不矛盾）
3. 如果更新了 decision-tree.md 中的节点状态，检查耦合矩阵中的关联节点
4. 向用户简要说明：更新了哪些文件、做了什么改动、对下一步的建议

## 注意事项

- 保持文档简洁，避免冗余
- 访存分析始终使用 memory-bound 假设（访存量 ≈ 性能）
- 关键参数：h（隐藏层维度）、n（输出维度，0.5h~4h）、N（基向量库大小，8h）、K（激活数）、θ（Kneedle 阈值）、τ（缩放系数）、p（补偿比例）
- 对用户的想法要诚实评估，不要回避指出问题
- 如果输入类型不明确，先问用户确认
- 新建 idea 文件时，在 research-log.md 头部的辅助文档链接区域添加引用

## 衔接实验实现

**重要**：当用户的输入涉及"实现实验"、"测试 idea"、"写代码验证"、"开始做实验"等意图时（即从规划阶段转入实现阶段），你**必须**在开始编写任何实验代码之前，先调用 `luturbo_experiment` skill 获取实验开发规范。

触发条件：
- 用户明确要求实现某个实验
- 用户说"先测试一下"、"写个脚本"、"跑一下看看"
- 当前任务需要在 `experiments/` 目录下创建或修改代码
- 实验规划完成后进入编码阶段

`luturbo_experiment` skill 提供：
- 数据集路径、LUT 权重路径等关键资源配置
- 代码结构和结果目录规范
- Shell 脚本模板
- Pre-Implementation Checklist（避免常见错误）
- Hookpoint 命名约定

**不要跳过这一步**。过去因为没有加载实验规范而导致了数据集路径错误、结果保存格式不对、hookpoint 映射问题等多个可避免的错误。
