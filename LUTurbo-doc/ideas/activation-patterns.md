# Idea: 激活模式分析 — C1 候选方案的 Oracle 上限评估

> 关联文档：[research-log.md](../research-log.md)（主文档）、[decision-tree.md](../decision-tree.md)（决策树）
> 涉及决策点：C1（选择算法）、A2b（条件分解）
> 状态：待验证

## 1. 核心想法

C1 有多个候选方案（增量更新、层次化选择、条件分解、分组选择等），但在投入实现之前，需要知道每个方案**即使在理想条件下**能带来多大收益。

本 idea 的方法论是：**用 oracle baseline 测上限，而非用描述性统计猜可行性。**

具体地，对每个 C1 候选方案，模拟其"拥有完美先知信息"时的效果——如果上限就不够好，方案可以直接排除；如果上限很好，再研究如何用低开销近似实现。

所有分析共享一个统一的评估框架：**候选集大小 → top-K recall → 重构误差**曲线。这使得不同 C1 方案可以直接在同一坐标系中比较。

## 2. 分析维度与 Oracle Baseline

### 2.1 增量选择（对应 C1e）

**核心问题**：保留前一 token 的选择结果，只替换少量基向量，能达到多高的 top-K recall？

**Oracle 实验**：
- 保留前一 token 的选择结果，允许替换 m 个位置（m = 0, 4, 8, 16, 32, 64, K）
- 替换时使用 oracle：从 N 个基向量中挑出最应该加入的 m 个（即真实 top-K 中不在旧集合里的，按激活值从大到小取前 m 个）
- 测量 oracle incremental top-K recall 和对应的重构误差
- **三种保留集变体**（测试不同记忆深度的上限）：
  - 保留 S_{t-1}（前一 token 的 top-K，K 个元素）
  - 保留前一 token 的 top-L 候选池（L = 1.5K, 2K）——真实系统做 top-K 时可顺手保留次优候选
  - 保留 S_{t-1} ∪ S_{t-2}（前两步的 top-K 并集）——测两步时序记忆的叠加价值

**为什么不只看重叠率**：
- 重叠率 |S_t ∩ S_{t+1}| / K 是均值统计，掩盖了分布信息。增量更新关心的是"每步要替换几个"的分布，尤其是**尾部**——哪怕平均只需替换 20 个，如果 5% 的 token 需要替换 80 个，增量方案就不稳定。
- Oracle recall 直接测量"允许替换 m 个时能恢复多少真实 top-K"，这才是 C1e 的生死线。

**假设与推测**：
- 相邻 token 的残差流具有局部连续性，线性编码器保持这种相似性，因此 top-K 集合变化缓慢。
- 深层重叠率可能高于浅层（特征更抽象、变化更缓慢）。
- Decode 阶段（自回归逐 token）的重叠率可能高于 prefill 阶段（不同位置并行处理），因为 decode 时每步只新增一个 token 的信息。

**补充观测**：
- 替换数 (K - |S_t ∩ S_{t+1}|) 的分布直方图，尤其关注 P90/P99 尾部
- 不同距离 d=1,2,4,8 的衰减，判断"时序记忆"范围
- Value-aware 版本：被替换掉的基向量的激活值大小分布——如果被替换的都是弱激活，漏掉的重构代价小
- **new-mass ratio**（主指标）：new_mass = Σ_{i ∈ S_t \ retained} |α_i| / Σ_{i ∈ S_t} |α_i|（retained 为保留集，可以是 S_{t-1}、top-L、或 S_{t-1}∪S_{t-2}）。比 overlap count 更直接地回答"新进来的基向量占了多大重构贡献"。如果 new-mass 普遍很小（< 10%），即使 overlap 不高，漏掉新基向量的代价也低。
- **burstiness**：连续多少个 token 都需要大替换量的 run length 分布。使用固定阈值（replacement > K/4, K/2）保证跨层跨变体可比，辅以相对阈值（> median+1σ）做单方案内部分析。工程上最怕连续坏段——连续延迟尖峰无法靠流水线掩盖，比散落的偶发大替换严重得多。

### 2.2 热集选择（对应新方案 C1h — 层次化选择）

**核心问题**：固定一个全局热集 H，每个 token 的 top-K 中有多少能被 H 命中？

**Oracle 实验**：
- 取全局激活频率最高的 |H| 个基向量作为热集（|H| = N×1%, 5%, 10%, 20%）
- 对每个 token，计算 **per-token hotset recall** = |S_t ∩ H| / K
- 关键不是全局覆盖率，而是 per-token recall 的**分布**——如果某些 token 热集命中率只有 10%，层次化对这些 token 无效

**为什么不只看频率分布/基尼系数**：
- 频率长尾只说明"有些基向量被很多 token 使用"，不等于"每个 token 都能从热集获益"。
- 极端反例：如果存在两种截然不同的 token 类型，各自有不同的热门基向量，那全局热集对任何一种类型的 per-token recall 都可能不高。
- Per-token hotset recall 直接测量"热集对这个具体 token 值多少"。

**假设与推测**：
- 语言模型的激活存在 Zipf-like 不均匀性，但这种不均匀性是否足以让一个小热集在 per-token 层面也有效，尚不确定。
- 如果 per-token recall 均值高（>40%）且方差小，层次化选择有价值；如果均值高但方差大（双峰分布），需要 conditional hotset（不同 token 类型不同热集），这就退化成 A2b。

**补充观测**：
- 热集中基向量的平均激活值 vs 非热集基向量的平均激活值——如果热集基向量系数也普遍较大，热集的重构贡献更大
- 剩余未命中部分的搜索空间有多大：如果热集命中了 50% 的 K，剩下 K/2 个需要从 N-|H| 个候选中搜索，搜索空间是否有意义地缩小

### 2.3 条件子库（对应 A2b 条件分解 / C1c 分组路由）

**核心问题**：如果先知道输入属于哪个"簇"，能把搜索空间缩到多小，同时保持多高的 top-K recall？

**Oracle 实验**：
- 对所有 token 的激活模式做离线聚类（G = 8, 16, 32, 64），聚类特征使用激活值加权的稀疏向量而非纯二值索引
- 为每个簇构建专属子库：该簇内所有 token 的 top-K 索引的并集（可能需要截断到 N_sub 大小）
- 测量 **oracle routing 下的 top-K recall**：假设每个 token 被完美路由到正确的簇，在子库内能覆盖多少真实 top-K
- 画 **子库大小 N_sub vs oracle recall** 曲线

**为什么不只看轮廓系数**：
- 轮廓系数在高维稀疏空间中对距离度量高度敏感，阈值不可靠。
- A2b 真正关心的不是"聚类有多紧致"，而是"子库有多小 + recall 有多高"。一个聚类即使轮廓系数不好看，只要子库显著小于 N 且 recall > 95%，A2b 就值得做。
- 还需要评估**路由错误的代价**：如果 token 被路由到错误的簇，recall 掉多少？如果掉得不多，说明簇间子库有重叠，路由不需要很精确。

**假设与推测**：
- 自然语言的语义类别（名词/动词/功能词）和句法角色可能在 SAE 空间中对应不同的激活模式子集，使得条件子库有效。
- 子库大小预期在 N/G 到 N 之间。如果接近 N/G（不同簇的激活几乎不重叠），条件分解收益巨大；如果接近 N（不同簇激活高度重叠），收益很小。
- MLP 层可能比 attention 输出层更有聚类结构（MLP 更像"特征存储"）。

### 2.4 种子扩展（对应新方案 C1i — 共激活引导选择）

**核心问题**：已知少量"种子"激活，利用共激活关系扩展候选集，需要多大的候选集才能覆盖真实 top-K？

**Oracle 实验**：
- 从真实 top-K 中取前 s 个最强激活作为种子（s = 4, 8, 16, 32）
- 构建共激活近邻表：对每个基向量 i，记录其 top-n 共激活伙伴（从训练数据统计）
- 扩展候选集：取所有种子的 top-n 共激活伙伴的并集
- 测量 **候选集大小 vs top-K recall** 曲线

**为什么不只看共激活矩阵的有效秩**：
- 低秩不等于可分组。共激活矩阵低秩可能只是因为频率不均匀（高频基向量和所有人共激活），而非真正的结构化依赖。
- 真正重要的是"能否从少量种子高精度恢复完整 top-K"——这直接测量共激活信息的预测价值。
- 如果种子 s=16（只需要搜索 16 个基向量的开销），候选集 < N/4 时 recall > 90%，说明共激活引导是一条可行路线。

**假设与推测**：
- 语义特征的组合性暗示某些基向量倾向于共同出现。但这种共激活是否足够稳定和精确（不只是统计噪声），需要实验验证。
- 风险：很多共激活可能是由语料频率和 token 类型混杂造成的（如高频基向量和所有其他基向量都有较高共激活概率），不一定能转化为有效的选择信号。构建近邻表时应控制基频（如用 PMI 而非原始条件概率）。

### 2.5 层间差异

**观测对象**：以上四个 oracle baseline 在不同 Transformer 层之间的变化。

这是唯一一个"描述性"维度，但它不增加系统复杂度（每层独立配置即可），且可能揭示：
- 某些层天然更适合某种 C1 方案（深层重叠率高 → 用 C1e，浅层有聚类 → 用 A2b）
- 某些层本身就很容易选择（top-K 集中在少量基向量），可以作为优先优化目标

## 3. 统一评估框架

所有 C1 方案最终都在做同一件事：**构造一个小于 N 的候选集，再从中精排**。因此统一用以下曲线比较：

```
X 轴: 候选集大小（或等效的访存量）
Y 轴: top-K recall（或重构 MSE / FVU）
```

每个 oracle baseline 对应曲线上的一系列点：
- C1e（增量）：候选集 = S_{t-1} + oracle 选的 m 个新候选，大小 = K + m
- C1h（热集）：候选集 = H + 冷集搜索范围，大小 = |H| + 搜索范围
- A2b（子库）：候选集 = 某簇的子库，大小 = N_sub
- C1i（种子扩展）：候选集 = 种子的共激活近邻并集，大小可变

在同一张图上画出四条曲线，可以直接看出哪种方案在相同候选集大小下 recall 最高。

### Value-aware 评估

除了 top-K recall（只看索引命中），还需要 value-aware 指标：
- **加权 recall**：recall_w = Σ_{i ∈ S_pred ∩ S_true} |α_i| / Σ_{i ∈ S_true} |α_i|。漏掉弱激活比漏掉强激活代价小。
- **重构 MSE**：用候选集内的基向量重构，分别测 C2a（内积系数）和 C2c（CG, t=5）两档。不使用完全 oracle 最优系数（完整最小二乘），以免高估 C1 方案的实际价值。
- **p 变化**：在 τ=0.3 下的超阈值维度比例 p，和补偿开销直接相关。

## 4. 数据采集的注意事项

### Token 类型区分
不能把所有 token 混在一起分析，至少分两组：
- **Decode 阶段 token**：对 CPU 低延迟推理场景更关键（自回归逐 token 生成）
- **Prefill 阶段 token**：可以 batch 并行，对延迟敏感度低一些

如果发现差异大，进一步按 token 类型（标点/功能词/内容词）细分。

### Value-aware 表示
分析对象不应只是二值索引集合 {i : i ∈ S}，而应包括激活值。两个 token 可能共享大量索引，但权重分布完全不同。至少补充：
- 加权 Jaccard（按激活值加权的集合相似度）
- 只统计 top-m 强激活（m < K，如 m=32）的模式，过滤掉弱激活噪声

### 数据分布
- 主实验：训练同分布数据
- 鲁棒性检查（可选，标注为 secondary）：不同风格语料（如换一个数据集），确认模式不是数据集 artifact

## 5. 与其他研究方向的关系

### 与 CG 系数（C2c）的协同
- CG 的效果取决于选中基向量的正交性。本分析中共激活模式间接反映正交性。
- 统一评估框架中的"重构 MSE"可以同时测 CG 系数和内积系数，作为 CG 实验的交叉验证。

### 与 D2（模型微调）的独立性
- 激活模式分析基于当前 SAE+LLM 的行为。D2 改变 LLM 后需重新分析。
- 但分析方法和脚本不变，基础设施可完全复用。

### 对决策树的更新
- 实验结果可能需要新增 C1h（层次化选择）、C1i（共激活引导选择）两个节点。
- 各 oracle baseline 的结果直接更新对应 C1 方案的"状态"列（可行/不可行/有条件可行）。

## 6. 补充分析（条件熵）

> **实现优先级：低**。仅在核心 oracle baseline 全部完成后，如需解释结果时再计算。不作为决策门槛，不值得为此增加显著实现成本。

作为辅助指标，计算以下条件熵（用于解释 oracle baseline 的结果，不作为独立判据）：
- H(S_t | S_{t-1})：给定前一 token 的选择后，当前 token 的选择不确定性。帮助解释 C1e oracle 结果好/坏的原因。
- H(S | cluster)：给定簇 id 后选择的不确定性。帮助解释 A2b oracle 结果。
- 对比 H(S) 和条件熵的差值，判断"条件信息能减少多少选择不确定性"。
