# Idea: CG 最优系数 — 解耦选择与系数计算

> 关联文档：[research-log.md](../research-log.md)（主文档）、[decision-tree.md](../decision-tree.md)（决策树）
> 涉及决策点：C2（系数计算）、B（基向量库训练）、C1（选择算法）
> 状态：已验证（Phase 1 完成）；当前不建议进入 Phase 2，作为备选保留
> 最新实验：[experiments/20260315-cg-coefficients-phase1.md](../experiments/20260315-cg-coefficients-phase1.md)

## 1. 核心想法

将基向量的"选择"和"系数计算"彻底解耦：
- 选择：由编码器（或其他轻量方法）决定激活哪 K 个基向量
- 系数：由 CG 迭代求解最优值，不依赖编码器输出

这个改变同时作用于训练和推理两个阶段。

## 2. 与当前方案的对比

### 当前方案（C2a：编码器内积系数）

```
训练：
  z = TopK(Ex + b)        // 编码器同时给出选择和系数
  x̂ = z^T D               // 解码器重构
  Loss = ||x - x̂||²       // 梯度同时更新 E 和 D

推理：
  z = TopK(Ex + b)        // 需要完整编码器（h × N 访存）
  x̂ = z^T D
```

编码器 E 身兼二职：选择 + 系数。训练信号混合了两者的误差。

### 新方案（C2-CG：CG 最优系数）

```
训练：
  z = TopK(Ex + b)                  // 编码器只负责选择
  S = nonzero_indices(z)            // 选中的 K 个索引
  B_S = D[S, :]                     // 取出选中的 K 个基向量，K × h
  α = CG_solve(x, B_S, α_init=z_S) // CG 求最优系数，初始值用编码器输出
  x̂ = B_S^T α                      // 用最优系数重构
  Loss = ||x - x̂||²                // 梯度更新 E 和 D

推理：
  S = selection(x)                  // 选择算法（可以不是编码器）
  B_S = D[S, :]
  α = CG_solve(x, B_S)
  x̂ = B_S^T α
```

## 3. 假设与当前证据

### 3.1 基向量库质量提升（训练时加入 CG 的预期收益）

原始动机是：当前 Loss 混合了选择误差和系数误差。CG 消除了系数误差后，解码器 D 收到的梯度信号可能更聚焦于"选对了吗"和"基向量本身好不好"。

Phase 1 的结果说明，这个方向在当前 SAE 上确实有理论收益，但上限不高：
- 保持选择不变时，CG(10 iter) = exact LS，说明系数优化本身已经被测到理论天花板
- MSE 下降约 10~17%，但只转化为 `p` 的绝对下降 1~3pp
- 选中基向量的条件数主要落在 5~25，存在相关性，但不属于严重病态

这意味着：`C2` 确实不是最优，但也不是当前主矛盾。即使训练时引入 CG，最乐观的收益也大概率落在一个已经不大的上限附近；而训练后若基向量更正交，CG 的边际收益还可能进一步下降。因此当前不建议进入 Phase 2。

### 3.2 编码器职责简化

编码器只需要做选择（分类任务），不需要同时给出精确系数（回归任务）。

可能的推论：
- 编码器可以更小（不需要 h × N 的满秩矩阵来做选择）
- 结构化编码器（稀疏、低秩、分块）在选择任务上可能表现够好
- 这为解决 C1（选择开销）问题创造了空间

目前这部分仍没有直接证据。Phase 1 只证明了：
- 编码器初始值能让 CG 收敛更快，说明它提供了有价值的方向先验
- 零初始化也能收敛到同一解，说明编码器并不是求最优系数的必要条件

但这还不足以推出"选择任务已经足够简单，轻量编码器就能解决 C1"。因此这条链路暂时只能作为间接启发，不能作为推进 CG 主线的理由。

### 3.3 训练-推理一致性

训练和推理都用 CG 系数，没有 mismatch。
当前方案如果推理时换用其他选择算法（比如 ANNS），选择结果和训练时编码器的选择不同，存在 mismatch。
CG 方案中，无论用什么选择算法，系数都是对选定基向量的最优解，更鲁棒。

这仍然是 CG 的概念性优点，但 Phase 1 表明：仅靠这个优点不足以覆盖其额外在线开销，因此不能单独构成主线推进理由。

## 4. 开销分析

### 4.1 推理时 CG 开销

```
约定：B_S ∈ R^{K×h}（K 个 h 维基向量作为行向量）
正规方程：B_S B_S^T α = B_S (x - b_dec)，其中 B_S B_S^T 是 K×K Gram 矩阵

每次 CG 迭代：
  v = B_S^T α  （K → h，重构，O(Kh) 计算，读 K 个基向量）
  w = B_S v    （h → K，投影回系数空间，O(Kh) 计算，基向量已在 cache）
  更新 α, r, p （O(K)，可忽略）

总计每次迭代：2Kh 计算，Kh 访存（首次迭代从内存读，后续 cache 命中）
```

t 次迭代的开销对比（K=128, h=1024）：

| 方法 | 计算量 | 访存量 | memory-bound? |
|------|--------|--------|---------------|
| 内积（当前） | Kh = 131K | Kh = 131K | 是 |
| CG t=3 | 6Kh = 786K | Kh = 131K | 是（cache 复用） |
| CG t=5 | 10Kh = 1.31M | Kh = 131K | 是（cache 复用） |
| CG t=10 | 20Kh = 2.62M | Kh = 131K | 是（cache 复用） |
| 完整最小二乘 | K²h = 16.8M | Kh = 131K | **否，compute-bound** |

理论上，CG 的访存量和内积法相同，额外成本主要来自 cache 内复用下的计算。

但 Phase 1 的系统结论是：即使这些计算大多发生在 cache 内，只要它们换来的 `p` 改善只有 1~3pp，整体性价比仍可能为负。因此这里的开销分析只能说明"CG 没有明显的额外 DRAM 访存灾难"，不能推出"实际部署一定划算"。

**前提**：K 个基向量（128 × 1024 × 4 bytes = 512KB）需要放进 L2 cache。
现代 CPU L2 cache 通常 1-2MB，512KB 可以放下。

### 4.2 训练时 CG 开销

训练时每个样本的计算（前向）：

```
编码器前向：h × N = 8h²（不变，仍然需要编码器做选择）
CG 求系数：20Kh（t=10 时，与 Phase 1 验证配置一致）

对比：
  编码器：8 × 1024² = 8.4M
  CG：    20 × 128 × 1024 = 2.6M
```

在已验证的 `t=10` 配置下，CG 额外增加约 31% 的前向计算量；若只用 `t=5`，额外计算约 15%，但当前没有足够证据表明 `t=5` 对训练时前向足够。

反向传播：需要讨论（见 §5）。

### 4.3 额外存储开销

推理时需要访问基向量 B_S（解码器的 K 行）。
当前推理 pipeline 已经需要访问预计算表 P_S（K × n），基向量 B_S（K × h）是额外的访存。

```
额外访存：K × h = 131K 元素
当前查表访存：K × n（n=h 时 = 131K）

CG 让系数计算访存翻倍：从 Kh 增加到 2Kh
但仍远小于原始 matmul 的 h×n
```

**前提说明**：此处对比的是"选择与系数解耦"场景下的系数计算开销。在当前方案（C1a + C2a）中，系数是编码器计算 z = Ex + b 的副产品，取 TopK 后系数已有，不需要额外的 Kh 访存。但 C1a 本身不可行（8h² 访存），因此实际推理必须使用某种轻量选择方法。在这个前提下：
- 内积系数需要额外读编码器 E 的 K 列（Kh）
- CG 系数需要额外读解码器 D 的 K 行（Kh）
- 两者的系数计算访存量相同

CG 的额外约束是：迭代过程中需要 B_S 持续驻留 cache。如果其间有其他内存访问导致 cache eviction，实际访存量会高于理论值。

## 5. 若未来重新启用该方向，需要回答的问题

### 5.1 反向传播怎么做？

CG 求解 α* 满足 B^T B α* = B^T x。需要计算 ∂Loss/∂D（更新基向量）。

**方法 A：隐式微分**
α* 是方程 B^T B α = B^T x 的解。由隐函数定理：
```
∂α*/∂B = -(B^T B)^{-1} · ∂(B^T B α* - B^T x)/∂B
```
不需要展开 CG 迭代，但需要求解另一个 K×K 线性方程组。

**方法 B：展开 CG 迭代（unrolled CG）**
把 t 次 CG 迭代视为计算图的一部分，直接用自动微分。
优点：简单，直接用 PyTorch autograd。
缺点：计算图变长，内存占用增加。

**方法 C：直通估计器（straight-through）**
对 α 的梯度直接传给编码器输出 z，忽略 CG 求解过程。
最简单但最粗糙。

如果未来真的重启 Phase 2，方法 B（展开 CG）仍然是实现成本最低的起点；但在当前结论下，这个问题优先级很低。

### 5.2 CG 收敛速度取决于什么？

CG 求解 B^T B α = B^T x 的收敛速度取决于 B^T B 的条件数。

如果选中的 K 个基向量彼此接近正交 → 条件数小 → CG 收敛快（可能 2-3 次迭代就够）。
如果基向量高度相关 → 条件数大 → CG 收敛慢（可能需要 10+ 次迭代）。

**有趣的推论**：如果训练时用 CG，Loss 会隐式地鼓励选出的基向量彼此正交（因为正交时 CG 收敛好，系数更优，Loss 更低）。这可能让基向量库的结构自然变得更好。

Phase 1 已测得当前 SAE 的条件数大多在 5~25。这个范围解释了为什么 CG 能稳定收敛，也解释了为什么它的收益存在但有限。

### 5.3 CG 初始值的影响

推理时 CG 的初始值选择：
- **零初始化**：无先验，需要更多迭代
- **编码器内积初始化**：如果推理时仍有编码器（或轻量替代），用其输出作为初始值，CG 只需微调
- **上一个 token 的系数**：如果相邻 token 激活相似（待验证），可以复用

初始值越好 → 需要的迭代次数越少 → 开销越低。

Phase 1 已验证：编码器初始值比零初始化收敛更快，但最终解相同。这个结论支持"编码器输出可作为 warm start"，但不足以改变整体 go/no-go 判断。

### 5.4 与选择算法（C1）的联动

如果选择算法给出的不是最优的 K 个基向量（比如用 ANNS 近似选择，recall < 100%），CG 能否补偿？

直觉：CG 对"选了哪些"给出最优系数。即使选择不完美，CG 也能在当前选择下尽量最小化误差。这比内积系数的容错性更好。

这仍然是值得保留的假设，但 Phase 1 没有直接验证，因为实验中选择始终固定为 SAE 编码器输出。

### 5.5 K 的下界是否会改变？

当前 K≈128 是用内积系数时的精度要求。如果 CG 系数更优，同样的精度目标下 K 是否可以更小？

如果 K 能从 128 降到 64 或 32：
- 查表访存减半或降到 1/4
- CG 每次迭代开销也减半或降到 1/4
- 选择空间虽然不变（仍从 N 个中选），但 ANNS 等方法的开销随 K 减小而降低

这是 CG 重新进入主线的最重要触发条件之一。但截至 Phase 1，还没有观察到支持该假设的直接证据。

## 6. 实验进度与后续策略

原始计划是三阶段推进；截至 2026-03-16，状态已更新如下：

| 阶段 | 当前状态 | 结果/判断 | 后续动作 |
|------|----------|-----------|----------|
| 1 | 已完成 | CG 系数相对内积系数有稳定精度收益，且 `10 iter = exact LS` | 结论成立，作为该方向的上界证据保留 |
| 2 | 暂停，不建议进入 | 当前 `p` 的绝对改善仅 1~3pp，难以覆盖额外在线计算；训练后 CG 收益还可能继续变小 | 除非出现新的强证据，否则不启动 |
| 3 | 暂停，不单独以 CG 为入口推进 | "编码器可简化" 仍是间接假设，当前更适合在 C1/B 主线中单独验证 | 转入 C1/B 主线而非 CG 主线 |

### 6.1 重新启用 CG 的触发条件

只有在以下情况之一出现时，才建议重新打开这条线：

1. `K` 显著降低到 `64` 或 `32`，使 CG 的相对开销明显下降
2. 新的 B/C1 方案下，Phase 1 风格评估显示 CG 能带来更大的 `p` 绝对下降（而不只是 MSE 改善）
3. C1 近似选择方案需要一个"给定候选集后的最优系数上界"作为分析工具，此时 CG 适合作为评估基线而非部署方案

### 6.2 当前建议

- 部署路径保持 `C2a`（内积系数）
- 研究主线优先转向 `C1`（选择策略）、`B`（更优 SAE/结构约束）、`D1-sub`（自适应阈值）
- CG 保留为备选和分析工具，不再作为近期主线突破口
